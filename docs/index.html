VLA Pretraining and Mid-Training Recipes

This page summarizes the main VLA pretraining and mid-training guidance: when to pretrain, when to mid-train, what data and action interfaces matter, and where Isaac-GR00T is useful as an implementation case study.

The focus is practical training content for vision-language-action models, including recipes, ablations, data quality checks, infrastructure requirements, and evaluation gates.

Training Taxonomy

StageDefinitionTypical dataIn scope
Generic VLM/LLM pretrainingBroad language, vision, and multimodal representation learning before robot grounding.Web text, image-text, video-text, general VQA.Only as initialization context.
VLA pretrainingTrain a model to map observations and language to action-relevant outputs across tasks or embodiments.Robot trajectories, human videos with latent actions, synthetic trajectories, auxiliary VLM data.Yes.
VLA mid-trainingBridge a pretrained VLM/VLA toward embodied action learning before narrow task adaptation.Embodied VQA, robot trajectories, spatial reasoning data, action-relevant videos.Yes.
Task adaptationSpecialize to a target robot or task after base VLA training.Target robot demonstrations.Contrast only.

Boundary rule: mid-training is not generic VLM pretraining because it is selected around embodied/action distributions, and it is not task fine-tuning because it should improve the broad initialization before committing to one task.

Pretraining Guidance

  • Use VLA pretraining when the target domain needs broad task, object, embodiment, or environment generalization that cannot be covered by a small downstream dataset.
  • Default to initialized VLM or VLA weights unless from-scratch provenance, architecture research, or data-control requirements justify the cost.
  • Gate scale-up on closed-loop benchmark improvements, not only lower action loss.
  • Evaluate base/pretrained behavior directly before claiming pretraining capability.

Mid-Training Guidance

VLA mid-training is a bridge stage that uses embodied, action-relevant, or robot-aligned data to adapt an initialized VLM/VLA before target task adaptation.

  • Use it when initialized VLM features are weak on early robot-action learning.
  • Prefer representation-preserving updates first: frozen backbone, LoRA, top-layer tuning, then full tuning only if justified.
  • Curate embodied/spatial/VLA-aligned data rather than adding generic VLM data blindly.
  • Measure early learning curves and held-out embodiment performance, not only final adapted task success.

Data Mixtures and Quality

  • Robot trajectories need synchronized images, states, actions, language, masks where available, and embodiment metadata.
  • Record action units, reference frames, gripper semantics, control frequency, horizon, cameras, and normalization statistics.
  • Audit mixture weights, corrupt samples, language quality, per-embodiment coverage, and benchmark leakage before scaling.
  • Use human videos or synthetic trajectories only when they supply action-relevant latents, future states, or physically plausible trajectories.

Architecture and Initialization

ComponentChoicesRecommendation
Vision-language backboneFrozen VLM, LoRA, top-layer tuning, full tuning, from scratch.Start with pretrained VLM and preserve useful representations unless evidence supports full tuning.
Connector/projectorRandom init, VLM projector reuse, action-conditioned connector.Train early and ablate whether it bottlenecks action learning.
Temporal moduleHistory, video encoder, recurrence/cache features.Add only when temporal disambiguation matters; validate latency.
Action headDiscrete tokens, latent actions, diffusion, flow matching, hybrid.Match action frequency and precision requirements.
Embodiment conditioningText prompts, soft prompts, IDs, per-embodiment modules.Required for heterogeneous robots; avoid one head per platform unless scale is limited.

Action Representation

RepresentationGood fitMain risk
Discrete action tokensAutoregressive VLM reuse and language/action unification.Poor tokenization damages precision and smoothness.
Latent actionsVideo pretraining without robot action labels.Latents may not decode cleanly to target robot actions.
Continuous diffusionMultimodal high-dimensional actions.Sampling latency and denoising stability.
Continuous flow matchingFast continuous action chunks.Needs careful horizon, timestep, and latency tuning.
Relative EEF actionsCross-embodiment human/robot transfer.Requires reliable frame conventions and relative statistics.

Large-Scale Training Infrastructure

  • Use streaming or sharded datasets with deterministic sampling and resumable provenance.
  • Track global batch size, gradient accumulation, precision, checkpoint policy, and actual distributed world size.
  • Save model, optimizer, scheduler, RNG, processor/tokenizer, dataset statistics, config, code commit, and data manifest.
  • Monitor loss, action loss by modality, data-loader latency, throughput, skipped/corrupt samples, and per-dataset sampling.
  • Do not scale until checkpoint resume, data mixture sampling, and closed-loop evaluation are validated.

Evaluation

LevelPurposeExamples

Open-loop metrics are smoke tests. They do not replace closed-loop success.

Minimum Experiments Before Scale-Up

  • Scratch versus initialized backbone on a small budget.
  • Frozen backbone versus LoRA versus top-layer tuning versus full tuning.
  • Discrete tokenizer versus continuous flow/diffusion on the same data.
  • Action horizon and observation/state history length.
  • Relative versus absolute action encoding where EEF actions are available.
  • With and without low-quality or synthetic data slices.
  • Closed-loop benchmark improvement, not just lower action loss.
